# digiwallsys — v1.0.0

> *Name: Anchor*

**Status:** Pre-release

---

## 📖 Overview

**Anchor** establishes the secure base of digiwallsys: account registration,
verified identity, wallet balances, provider-backed funding intents, atomic
peer-to-peer transfers, transaction history, and a PostgreSQL financial core.

The release replaces the original caller-controlled wallet operations with
server-derived identities, short-lived access tokens, rotating refresh sessions,
idempotent writes, risk checks, and balanced immutable journals. It also turns
the original Expo prototype into a documented monorepo with migrations, CI,
Docker, tests, release automation, and a production-bundle check.

---

## 🎯 Objectives

- Establish a stable incoming/outgoing payment foundation.
- Ensure every money movement is atomic, balanced, idempotent, and auditable.
- Remove generated demo balance credits from user-facing API flows.
- Protect sessions with verification, recovery, rotation, lockouts, and biometrics.
- Provide a reproducible PostgreSQL schema, test strategy, and release process.
- Preserve the exact tag `v1.0.0` and exact release name `Anchor` while marking
  the GitHub release as a pre-release.

## ✨ Added

### 📒 Double-entry financial core

- Immutable `ledger_journals` and `ledger_entries` tables.
- Balanced debit and credit enforcement through deferred PostgreSQL triggers.
- Wallet-liability and provider-clearing accounts.
- Administrator reconciliation comparing cached balances with ledger balances.
- Stable wallet row locking for concurrent transfers.

### 🏦 Provider-verified funding

- Idempotent funding intents with provider references.
- HMAC-verified webhooks and provider-event deduplication.
- Wallet credit only after a verified successful event.
- Funding status history and in-app success/failure notifications.

### 🔐 Account and session security

- Email verification and password-reset tokens with hashed storage and expiry.
- Fifteen-minute access tokens and rotating opaque refresh tokens.
- Refresh-token replay response and session-family revocation.
- Login failure counters and temporary account lockout.
- Expo SecureStore token persistence and biometric session unlock.

### 🛡️ Transaction safety

- Required idempotency keys for transfers, funding, requests, and schedules.
- Single-transfer, hourly-velocity, and daily-amount fraud controls.
- API and authentication-specific rate limits.
- Audit logging for security, funding, transfer, fraud, and reconciliation events.

### 📱 Wallet experience

- Balance, verified funding, send, history, receipts, and CSV export screens.
- Payment requests and QR request generation/scanning.
- One-time and recurring scheduled transfers.
- In-app notifications, spending alerts, Expo push registration, and preferences.
- Role-protected administrator overview, fraud review, audit log, and reconciliation.

### 🧪 Repository and quality

- Versioned SQL migrations and a migration runner.
- Disposable PostgreSQL integration and concurrent-overdraft tests.
- Backend smoke tests, mobile configuration tests, lint, and Android bundle export.
- Docker Compose, separate GitHub workflows, issue templates, and professional docs.

## 🏛️ Architecture progress

**Ledger-first transaction processing.** A transfer locks both wallets in stable
user order, runs risk checks, posts a balanced journal, updates cached balances,
records the transaction, creates notifications, and writes an audit event inside
one database transaction.

**Provider trust boundary.** Funding intent creation never changes a wallet.
Only a signed, deduplicated provider event can post the clearing-to-wallet ledger
journal and cached balance credit.

**Rotating session model.** Access tokens are short-lived. Refresh tokens are
opaque, hashed in PostgreSQL, rotated at every use, and revoked on logout or
password reset. The mobile client stores them with SecureStore.

Known limitations at this pre-release:

- The existing private hosted database credential is rejected by its provider.
- Local PostgreSQL integration tests require Docker/PostgreSQL, which is not
  installed in the current development environment; CI provides PostgreSQL 16.
- Real funding checkout, email delivery, and Expo push require provider credentials.
- Regulatory, KYC/AML, privacy, penetration-testing, and independent financial
  audit gates remain mandatory before any real-money use.
